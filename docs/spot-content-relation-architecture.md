# Spot-Content Relation Architecture

## 문서 목적

이 문서는 `하나의 스팟에 여러 작품이 연결되는 구조`를 장기적으로 안정적으로 다루기 위한 아키텍처 초안이다.

예를 들어 `도쿄타워` 같은 실제 장소는 여러 애니, 영화, 드라마, 게임, 아티스트 이벤트와 동시에 연결될 수 있다. 단순히 `spot.relatedContent[]` 형태로 작품 목록만 늘려 붙이는 방식은 데이터 설명력과 UI 확장성에 한계가 있다.

이 문서는 Kiro에서 Spec을 작성하기 위한 상위 설계 기준 문서로 사용한다.

## 문제 정의

현재 단순한 `Spot -> RelatedContent[]` 구조만으로는 다음 문제를 해결하기 어렵다.

- 한 스팟에 수많은 작품이 연결될 때 우선순위가 없다.
- 작품과 장소가 왜 연결되는지 설명하기 어렵다.
- `장면 등장`, `촬영지`, `이벤트 개최`, `굿즈 판매`, `팬 추정 성지`를 같은 강도로 표현하게 된다.
- 일시적 이벤트와 반영구적 연결을 구분하기 어렵다.
- 운영/검수 단위가 스팟 중심으로만 남아 관계 데이터 품질이 무너지기 쉽다.
- UI에서 관련 작품이 많아질수록 정보가 납작해지고 탐색성이 떨어진다.

## 핵심 방향

장기적으로는 `Spot`과 `Content`를 직접 배열로 연결하는 방식이 아니라, 둘 사이의 `관계(Relation)` 자체를 독립 엔티티로 승격해야 한다.

핵심 개념은 다음과 같다.

- `Spot`: 현실 장소
- `Content`: 작품
- `SpotContentRelation`: 장소와 작품의 연결 정보
- `RelationEvidence`: 해당 연결을 뒷받침하는 장면, 출처, 이벤트 정보
- `ExperienceLayer`: 방문자 관점의 부가 정보

즉 시스템이 저장해야 하는 것은 단순히

`도쿄타워는 명탐정 코난과 관련 있음`

이 아니라,

`도쿄타워는 명탐정 코난의 특정 장면/배경/이벤트 맥락으로 연결되며, 그 근거와 신뢰도와 기간은 무엇인가`

여야 한다.

## 목표

- 스팟 하나에 수십 개 작품이 연결되어도 데이터 구조가 무너지지 않아야 한다.
- 작품-장소 연결의 `맥락`, `근거`, `신뢰도`, `기간성`을 표현할 수 있어야 한다.
- UI에서 대표작, 관계 유형, 검증 수준에 따라 정렬/필터링이 가능해야 한다.
- 운영자가 스팟이 아니라 `관계` 단위로 검수할 수 있어야 한다.
- 커뮤니티가 `새 스팟 추가`뿐 아니라 `새 관계 추가`, `근거 보강`, `기간 종료 처리`에 기여할 수 있어야 한다.

## 비목표

- 단기적으로 모든 기존 기능을 전면 재작성하는 것
- 초기 단계부터 지나치게 복잡한 편집 UI를 도입하는 것
- 작품과 장소 외의 모든 엔티티를 한 번에 일반화하는 것

## 도메인 모델

### 1. Spot

현실 세계의 고유한 장소를 나타낸다.

예시:

- 도쿄타워
- 세이부돔
- 특정 역
- 특정 카페

책임:

- 위치 정보
- 기본 설명
- 사진
- 장소 상태
- 방문 편의 정보

`Spot`은 실제 장소의 정체성을 관리하며, 작품과의 연결 자체를 설명하는 책임은 갖지 않는다.

### 2. Content

작품 또는 IP 단위를 나타낸다.

예시:

- 애니메이션
- 영화
- 드라마
- 게임
- 아티스트
- 스포츠팀

책임:

- 제목
- 타입
- 대표 이미지
- 발매/방영 연도
- 기본 메타데이터

### 3. SpotContentRelation

이 구조의 핵심 엔티티다. `Spot`과 `Content` 사이의 연결을 표현한다.

예시:

- 도쿄타워 <-> 명탐정 코난
- 도쿄타워 <-> 러브라이브
- 도쿄타워 <-> 기간 한정 콜라보 이벤트

이 엔티티는 다음 질문에 답할 수 있어야 한다.

- 왜 이 작품이 이 장소와 연결되는가
- 이 연결은 얼마나 강한가
- 공식 정보인가, 커뮤니티 추정인가
- 현재도 유효한가
- 사용자에게 먼저 보여줘야 하는 관계인가

권장 필드:

- `id`
- `spotId`
- `contentId`
- `relationType`
- `summary`
- `confidenceLevel`
- `officialness`
- `displayPriority`
- `verificationScore`
- `sourceCount`
- `startDate`
- `endDate`
- `status`
- `createdBy`
- `updatedBy`
- `createdAt`
- `updatedAt`

### 4. RelationEvidence

관계를 뒷받침하는 구체적 증거다. 장면 이미지, 화수, 공식 링크, 이벤트 포스터, 비교 사진, 사용자 제보 등을 저장한다.

이 엔티티는 `Spot`에 직접 붙기보다 `SpotContentRelation`에 귀속되는 것이 더 정확하다.

예:

- `도쿄타워 <-> 작품 A` 관계의 장면 캡처
- `도쿄타워 <-> 작품 B` 관계의 공식 이벤트 공지 링크
- `도쿄타워 <-> 작품 C` 관계의 팬 비교 사진

권장 필드:

- `id`
- `relationId`
- `evidenceType`
- `title`
- `description`
- `imageUrl`
- `sourceUrl`
- `episodeInfo`
- `capturedAt`
- `submittedBy`
- `verificationStatus`
- `createdAt`
- `updatedAt`

### 5. ExperienceLayer

실제 방문자에게 유용한 정보 계층이다. 이 정보는 장소 중심일 수도 있고, 특정 관계에 종속될 수도 있다.

예:

- 이 작품의 장면과 비슷한 촬영 각도
- 콜라보 기간에만 열린 굿즈 존
- 방문 추천 시간대
- 야경 기준 포토스팟
- 이벤트 종료 여부

초기에는 별도 엔티티로 분리하지 않아도 되지만, 장기적으로는 relation 기반 확장이 가능해야 한다.

## Relation Type 설계

관계는 모두 같은 종류가 아니므로 최소한 유형을 구분해야 한다.

권장 예시:

- `scene_depicted`: 작품 장면에 직접 등장
- `inspired_by`: 장소를 모티프로 사용
- `filming_location`: 실사 촬영지
- `collaboration_event`: 콜라보 행사 개최
- `merchandise_spot`: 굿즈/전시/판매 장소
- `fan_inferred`: 커뮤니티 추정 성지
- `promotional_reference`: 홍보용 비주얼/포스터에 등장

이 구분이 없으면 모든 연결이 같은 강도로 노출되어 품질이 급격히 나빠진다.

## 신뢰도와 공식성

관계에는 `연결 강도`와 `데이터 출처 수준`이 모두 필요하다.

예시 필드:

- `confidenceLevel`
  - `high`
  - `medium`
  - `low`
- `officialness`
  - `official`
  - `community_verified`
  - `user_submitted`
  - `unverified`

예시 해석:

- 공식 이벤트 페이지가 있는 콜라보는 `official`
- 장면 비교와 다수 검증이 있는 배경 성지는 `community_verified`
- 제보만 있고 근거가 약하면 `user_submitted` 또는 `unverified`

## 기간성 모델

유명 랜드마크는 영구 관계와 기간성 관계가 섞여 있다. 이를 구분하지 않으면 오래된 팝업/이벤트 정보가 현재 정보처럼 남는다.

예:

- 작품 배경으로 등장: 장기 또는 반영구
- 콜라보 카페: 기간 한정
- 시즌 전시: 기간 한정
- 굿즈 판매 부스: 기간 한정

따라서 `SpotContentRelation`에는 기간 필드가 필요하다.

- `startDate`
- `endDate`
- `status`

`status` 예시:

- `active`
- `expired`
- `scheduled`
- `archived`

## 대표 관계와 정렬 우선순위

작품이 많은 스팟일수록 전체 나열보다 `대표성`이 중요하다. 장기적으로는 작품 자체가 아니라 관계를 정렬해야 한다.

권장 우선순위:

1. 운영자 수동 우선순위 (`displayPriority`)
2. 공식성 (`officialness`)
3. 증거 수 (`sourceCount`)
4. 검증 점수 (`verificationScore`)
5. 현재 유효성 (`status`, `date window`)
6. 사용자 반응 (조회, 체크인, 저장 등)

이 정렬을 기반으로 대표 관계 3개 정도를 헤더 근처에 먼저 보여주는 구조가 적합하다.

## 장기 UI 구조

스팟 상세 페이지는 단순한 `관련 작품 목록`이 아니라 `관계 해석 화면`이 되어야 한다.

권장 IA:

1. 스팟 기본 정보
2. 대표 관계 3개
3. 전체 관계 수 및 필터
4. 관계 유형별 보기
5. 관계별 근거/장면/이벤트 정보
6. 방문자 실전 정보
7. 같은 작품의 다른 스팟

권장 UI 요소:

- `관련 작품 18개` 같은 카운트 표시
- 대표 관계 카드
- 관계 유형 필터
- `장면 기반`, `이벤트 기반`, `팬 추정` 탭 또는 칩
- `더보기`/`접기`
- 만료 관계 숨김 또는 보관 처리

핵심 원칙:

- 작품이 많아도 한 화면에 무작정 펼치지 않는다.
- 대표성과 근거를 먼저 보여준다.
- “왜 이 작품이 여기와 관련 있나”를 한 번에 이해할 수 있어야 한다.

## 검색 및 탐색 구조

장기적으로 검색은 단순히 `Spot`과 `Content` 두 축만으로는 부족하다. `Relation`까지 포함한 탐색 모델이 필요하다.

검색 축:

- 장소 검색: `도쿄타워`
- 작품 검색: `명탐정 코난`
- 맥락 검색: `촬영지`, `야경 배경`, `콜라보 이벤트`

예상 흐름:

- 사용자가 `도쿄타워`를 검색하면
  - 스팟 결과
  - 관련 대표 작품
  - 현재 활성 이벤트 관계
- 사용자가 `명탐정 코난`을 검색하면
  - 작품 결과
  - 해당 작품과 연결된 스팟
- 사용자가 `도쿄타워 나온 애니`처럼 검색하면
  - relation 해석 기반 결과

## 운영 및 검수 구조

장기적으로는 검수도 `스팟 단위`와 `관계 단위`를 분리해야 한다.

권장 검수 단위:

- `Spot` 승인/수정
- `SpotContentRelation` 승인/보류/반려
- `RelationEvidence` 승인/보류/반려

예시 시나리오:

- 도쿄타워 스팟은 이미 승인됨
- 새 작품 연결은 검수 대기
- 연결 근거는 부족해서 보류
- 이벤트는 종료되어 상태 변경 필요

이 구조가 있어야 유명 장소에 대한 과등록과 오래된 관계 누적을 제어할 수 있다.

## 커뮤니티 기여 모델

장기적으로 커뮤니티 기여는 `스팟 추가`보다 `관계 추가/수정/검증` 비중이 커질 가능성이 높다.

예:

- 이 작품도 이 장소와 관련 있음
- 이 화수 장면을 근거로 추가
- 이 이벤트는 종료됨
- 이 관계는 공식이 아니라 팬 추정에 가까움

따라서 기여 UI도 장기적으로는 다음 액션을 지원해야 한다.

- 관계 추가
- 관계 근거 추가
- 관계 유형 수정 제안
- 기간 종료 신고
- 관계 신뢰도 투표

## 기존 구조와의 관계

현재의 `spot.relatedContent[]`는 단기적으로는 유지 가능하지만, 장기적으로는 `SpotContentRelation`의 축약 표현 또는 캐시된 뷰로 보는 편이 좋다.

즉 최종적인 정규 구조는 다음과 같다.

- `Spot`
- `Content`
- `SpotContentRelation`
- `RelationEvidence`

그리고 현재 UI의 `relatedContent`는 장기적으로 relation 기반 집계 결과로 대체한다.

## 점진적 마이그레이션 전략

### 1단계: 관계 엔티티 도입

- 기존 `Spot`, `Content` 구조는 유지
- `SpotContentRelation` 테이블/컬렉션 추가
- 기존 `relatedContent[]` 데이터를 relation으로 이관 가능하게 준비

### 2단계: 읽기 경로 전환

- 스팟 상세에서 관련 작품을 relation 기반으로 조회
- 대표 관계 정렬 로직 추가
- 단순 배열 대신 관계 메타데이터 일부를 활용

### 3단계: 근거 데이터 연결

- 장면/이벤트/링크를 relation에 귀속
- `왜 연결되었는지`를 UI에 표시

### 4단계: 검수 체계 분리

- 관계 승인 상태 도입
- 만료 이벤트/보류 관계 처리

### 5단계: 편집 및 기여 확장

- 관계 추가/수정/근거 보강 UI 도입
- 커뮤니티 검증 흐름 도입

## 기대 효과

- 유명 랜드마크에 작품이 많이 몰려도 데이터 구조가 무너지지 않는다.
- 관련 작품 수가 많아져도 대표성과 근거 중심으로 UX를 유지할 수 있다.
- 이벤트형 연결과 장면형 연결을 분리해 현재성을 보장할 수 있다.
- 운영자가 `스팟`과 `관계`를 구분해 관리할 수 있다.
- 이후 추천, 검색, 큐레이션, AI 보조 분류에도 유리한 기반이 된다.

## 리스크

- 초기 설계 및 운영 복잡도가 증가한다.
- 입력 폼과 관리자 도구 요구사항이 커진다.
- 기존 단순 구조에서 데이터 마이그레이션 비용이 발생한다.

하지만 장기적으로는 이 복잡도를 도입하지 않으면 허브형 스팟이 늘어날수록 데이터 품질과 UI 품질이 동시에 하락한다.

## 결론

장기적으로 가장 좋은 구조는 `장소`와 `작품`을 직접 배열로 묶는 모델이 아니다. `관계`를 1급 엔티티로 승격시키고, 그 관계에 `유형`, `근거`, `신뢰도`, `공식성`, `기간성`, `검수 상태`를 부여하는 구조가 필요하다.

이 문서를 기준으로 이후 Kiro Spec에서는 최소한 다음 범위를 독립적으로 구체화해야 한다.

- 도메인 모델 정의
- 저장 스키마 설계
- 조회 API 설계
- 상세 UI 정보 구조
- 운영/검수 플로우
- 점진적 마이그레이션 계획
